# Template-Only Thread Rules (SnapFit)

## 목적
- 이 스레드는 템플릿(커버/페이지) 작업만 진행한다.
- 결제/주문/알림/인증 이슈는 이 스레드에서 처리하지 않는다.
- 자동 생성형 템플릿 작업은 [/Users/devsheep/SnapFit/SnapFit/docs/TEMPLATE_AUTO_GENERATION_SYSTEM.md](/Users/devsheep/SnapFit/SnapFit/docs/TEMPLATE_AUTO_GENERATION_SYSTEM.md) 를 기준 명세로 사용한다.

## 기본 캔버스 규격 (고정)
- `COVER_SQUARE`: `1440 x 1440`
- `COVER_PORTRAIT`: `1080 x 1440`
- `COVER_LANDSCAPE`: `1440 x 1080`
- 페이지도 동일 비율 규격을 사용한다.

## 필수 규칙
1. Figma 원본 좌표/폰트/색상/레이어 순서를 우선한다.
2. 운영 기준 원본은 `portrait master(1080x1440)` 1세트를 우선한다.
3. `square/landscape`는 기본적으로 앱/JSON `variants`로 파생하고, QA에서 깨질 때만 비율별 Figma 원본 프레임을 추가 제작한다.
4. `strictLayout=true`, `autoFit=false`는 기본값이지만, 비율 파생 안정성이 더 중요할 경우 `variants` 보정값을 우선 적용한다.
5. 템플릿은 `cover + pages(12~24)` 세트로만 등록하되, 운영 권장값은 `14~20 페이지`로 본다.
6. 미리보기와 실제 적용(스토어 상세/스텝2/편집기/읽기화면)이 동일해야 한다.
7. 커버와 페이지는 동일 레이아웃을 반복하면 안 된다. 페이지별 역할과 구성이 명확히 달라야 한다.
8. 모든 템플릿 시안에는 예시 이미지가 반드시 들어가야 한다.
9. 예시 이미지는 무료이며 저작권 이슈가 없는 소스만 사용한다.
10. 사용자가 별도로 요청하지 않아도, 템플릿 작업은 기본적으로 `레이아웃 제작 -> 예시 이미지 삽입 -> 미리보기 검수`까지 이어서 완료한다.
10-1. 자동 생성형 템플릿은 기본적으로 `Figma JSON 파싱 -> token 추출 -> layout pattern 적용 -> ratio variant 생성 -> 예시 이미지 삽입 -> duplicate 검증 -> preview 검수` 순서로 진행한다.
11. 플레이스홀더 색 박스나 빈 이미지 슬롯 상태로는 “완료”로 간주하지 않는다.
12. 이미지 슬롯은 최소한 커버와 대표 페이지들에 실제 예시 이미지가 들어가 있어야 하며, 가능하면 전체 페이지 슬롯을 실제 예시 이미지로 채운다.
13. 예시 이미지는 템플릿 무드와 지역/주제에 맞는 사진으로 선택한다. 임의의 무관한 이미지 재사용은 금지한다.
14. Figma에서 최종 확정된 이미지는 배포 전에 반드시 export 해서 CDN에 업로드해야 한다.
15. 최종 템플릿 JSON에는 `figma.com/api/mcp/asset/...` 를 직접 저장하지 않는다.
16. 최종 이미지 URL은 직접 접근 가능한 CDN 공개 URL이어야 하며 404/핫링크 차단이 없어야 한다.
17. 로컬 asset은 fallback 또는 개발 검수용으로만 유지한다.
18. 커버에는 배경 이미지 또는 배경 비주얼이 반드시 포함되어야 하며, 첫 화면에서 템플릿의 무드가 즉시 전달되어야 한다.
19. 한 세트 안의 페이지는 정보 카드형만 반복하지 말고 `풀블리드`, `포토 프레임`, `타이포 중심`, `정보 정리형`, `엔드 포스터형` 등으로 역할이 분산되어야 한다.
20. 예시 이미지는 가능한 한 페이지마다 중복 없이 사용하고, 중복이 필요할 때는 크롭/역할이 명확히 달라야 한다.
21. 템플릿 톤앤매너는 한 세트 안에서 일관되어야 한다. 배경색, 타이포, 포인트 컬러, 사진 무드는 같은 아트디렉션으로 묶여야 한다.
22. 스토어 카드/상세 커버/페이지 미리보기는 가능하면 Figma 승인 export 이미지를 직접 사용하고, `templateJson` 재렌더는 fallback 으로만 쓴다.
23. `coverImageUrl` 과 `previewImages[0]` 는 반드시 동일해야 한다.
24. 페이지 미리보기용 `previewImages` 는 최소 4장 이상 유지하고, 대표 페이지가 실제 운영 이미지로 검수 가능해야 한다.
25. Figma export 후 업로드된 PNG/JPG 파일은 배포 전에 실제 디코드 검증을 통과해야 한다. 깨진 파일은 업로드 금지다.
26. 템플릿 생성 시 “피그마와 비슷한 레이아웃”이 아니라 “피그마 승인본과 동일한 구조/이미지/순서”를 목표로 한다. 차이가 있으면 Figma 기준으로 수정한다.

## 퀄리티 기준
- 첫 화면만 봐도 템플릿의 주제와 감정이 읽혀야 한다.
- 커버는 단순 텍스트 카드가 아니라 배경 이미지, 오버레이, 타이포 위계가 함께 설계된 완성형 포스터여야 한다.
- 포토 페이지는 사진이 실제로 보이는 수준의 존재감을 가져야 하며, 장식용 썸네일처럼 보이면 안 된다.
- 레이아웃은 안전한 반복보다 변주를 우선한다. 같은 카드형 박스 반복만으로 세트를 채우지 않는다.
- 텍스트만 있는 페이지도 허용하지만, 전체 세트 안에서 리듬을 만드는 역할이어야 한다.
- 페이지를 넘길수록 정보, 무드, 시선 흐름이 바뀌어야 하며, 모든 페이지가 같은 밀도와 같은 구조를 가지면 안 된다.
- 배경, 사진, 도형 마스크, 여백, 타이포가 함께 작동해서 “템플릿처럼 보이는” 수준이 아니라 “상품처럼 보이는” 수준까지 끌어올린다.

## QA 게이트 (등록 전)
- 텍스트 오버플로우 없음
- `template_parity_rules.json` 기준 줄바꿈/커버 간격/금지 배경 규칙 통과
- 이미지 404 없음
- 터치 선택/이동 불가 영역 없음
- 미리보기 == 실적용 일치
- 페이지별 레이아웃 다양성 확인
- 커버와 페이지 레이아웃 중복 없음
- 예시 이미지가 실제 시안에 모두 반영됨
- 대표 프레임뿐 아니라 핵심 사진 슬롯들이 실제 예시 이미지로 채워졌는지 확인
- Figma 승인 이미지와 CDN 배포 이미지가 동일한지 확인
- 최종 JSON에 Figma MCP asset URL이 남아 있지 않은지 확인
- `portrait master` 기준 `variants(square/landscape)` 파생 결과 확인
- 무료/저작권 안전 이미지 소스 사용 여부 확인
- 커버 배경 이미지/비주얼 존재 확인
- 포토 페이지의 사진 가시성 확인
- 전체 세트의 톤앤매너 일관성 확인
- 스토어 카드 == 상세 커버 == 페이지 미리보기 대표 이미지 일치 확인
- 업로드된 CDN 이미지가 실제 디코드 가능한 파일인지 확인
- release gate에서 Figma 정합 hard fail 이 없어야 함

## 작업 순서
1. Figma에서 `portrait master` 커버 + 페이지 프레임 제작
2. 커버 배경 비주얼, 예시 이미지 반영, 페이지별 레이아웃 다양화
3. 플레이스홀더 슬롯이 남지 않도록 실제 예시 이미지까지 채우고 Figma 시안 검수
4. Figma 이미지 export 및 CDN 업로드
5. handoff JSON 생성
6. `variants(square/landscape)` 생성 또는 보정
7. 로컬 변환 스크립트 실행
8. 앱에서 4개 화면(스토어 상세/스텝2/편집기/읽기화면) 검수
9. 필요 시 비율별 예외 Figma 프레임 추가
10. 관리자 업서트 및 서버 반영
11. 서버 반영 후 스토어 카드/상세 커버/페이지 미리보기에서 동일 이미지가 보이는지 최종 확인
